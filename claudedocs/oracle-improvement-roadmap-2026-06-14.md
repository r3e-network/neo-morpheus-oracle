## THE SINGLE HIGHEST-LEVERAGE CHANGE

**B1 — Stop finalizing transient worker HTTP 5xx/429 as permanent on-chain failure callbacks** (`fulfillment.js`).

This is the only **confirmed-high, code-fixable-here, money/callback-path correctness bug** with no infra dependency. Today a momentary enclave overload, an upstream 429, or a transient 502/503/504 is irreversibly burned into a hard `fulfillRequest(success=false)` to the consumer contract — terminal, never re-scanned — while the _identical_ error at the network layer would be retried. It silently corrupts the core product guarantee (correct callback delivery) on every lane (price/VRF/compute/confidential), and the fix is `small` + strictly-safer. Do this first.

**Critical correction from the verdict (the fix as originally written is subtly wrong):** `classifyError` does **not** treat the word "transient" as a keyword (I confirmed: its transient branch matches only `etimedout/econnrefused/econnreset/'rate limit'/'socket hang up'/503/502/504/network/'timed out'/unavailable`). So a thrown `"...(transient)"` for status **429/408/425/500/0** classifies as `unknown`, not `transient` — it only happens to retry because the fresh-event path retries everything non-`permanent`. **Implement by extending `classifyError`'s transient branch to recognize the retryable status codes (429/408/425/500/0) explicitly**, not by relying on the non-matching keyword. Keep deterministic 4xx (400/401/403/404/409/422) on the deliver-as-failure path. Apply the same gate to the `/oracle/decrypt` branch (`fulfillment.js:682`).

- **Gate:** new unit test — 503 **and** 429/500 retry; 400 finalizes `success:false`. Existing `relayer.test.mjs:842` (status 400 → success:false) must stay green. `npm run test:relayer`.
